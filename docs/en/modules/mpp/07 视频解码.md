<a name="ZH-CN_TOPIC_0000002441699449"></a><a name="ZH-CN_TOPIC_0000002408260194"></a><a name="ZH-CN_TOPIC_0000002408260210"></a><a name="ZH-CN_TOPIC_0000002441699409"></a><a name="ZH-CN_TOPIC_0000002441659569"></a>## Important Concepts - Stream Sending Mode The VDEC decoder provides three stream sending modes: - Stream Mode (OT\_VDEC\_SEND\_MODE\_STREAM): Users can send a stream of any length to the decoder each time, and the decoder internally completes the identification process for one frame of stream. Note that for H.264/H.265/MPEG4, the end of the current frame stream can only be identified upon receiving the next frame stream. Therefore, in this mode, after inputting one frame of H.264/H.265/MPEG4 stream, image decoding cannot be expected to start immediately. - Frame Mode (OT\_VDEC\_SEND\_MODE\_FRAME): Users send one complete frame of stream to the decoder each time. Each call to the send interface signals to the decoder that the frame stream has ended, and it begins decoding the image. Therefore, the stream sent in each call of the send interface must be a complete frame, otherwise decoding errors will occur. This mode enables fast decoding. - Compat Mode (OT\_VDEC\_SEND\_MODE\_COMPAT): Supports sending one frame stream to the decoder in multiple calls, but the end-of-frame flag end\_of\_frame must be set to TD\_TRUE at the end of each frame stream; otherwise, the current frame stream is considered not yet complete. The stream sending mode can be set in the [ss\_mpi\_vdec\_create\_chn](#ZH-CN_TOPIC_0000002408260210) interface. - Image Output Order According to the H.264/H.265/MPEG4 protocols, decoded images may not be output immediately after decoding. The VDEC decoder can achieve faster output by setting different image output modes. There are two image output modes: - Decoding Order: Decoded images are output in the order they are decoded. - Display Order: Decoded images are output according to the H.264/H.265/MPEG4 protocol. According to the H.264/H.265/MPEG4 protocols, the decoding order of a video is not necessarily the same as its output (i.e., display) order. For example, when decoding B-frames, both preceding and following P-frames are needed as references, so P-frames after the B-frame are decoded before the B-frame, but the B-frame is output before those P-frames. Output in decoding order is a prerequisite for ensuring fast output. If the user chooses decoding order output, they must ensure that the decoding order of the stream is the same as the display order. Combining frame-mode stream sending with decoding order output achieves fast decoding and fast output. Users must ensure that each sent stream is a complete frame and that the decoding order matches the display order. The image output mode out\_order can be set in the [ss\_mpi\_vdec\_set\_chn\_param](#ZH-CN_TOPIC_0000002441699409) interface. - Timestamp (PTS) Handling When sending streams in OT\_VDEC\_SEND\_MODE\_FRAME mode, the PTS of the decoded output image is the PTS provided by the user in the stream sending interface ([ss\_mpi\_vdec\_send\_stream](#ZH-CN_TOPIC_0000002441699449)). The decoder will not modify this value. If the configured PTS value is 0, it means the user is not performing frame rate control; instead, the video output module (VO) will perform frame rate control. When VDEC is in playback mode and VPSS is in AUTO mode, if the user-provided PTS value is -1, it means this image will not be displayed by the video output module (VO). If it is any other value, the video output module (VO) performs frame rate control based on the PTS value set by the user. **Note: PTS values of 0 and non-zero must not be mixed.** When sending streams in OT\_VDEC\_SEND\_MODE\_STREAM mode, the PTS of all decoded output images is uniformly set to 0, indicating that the user is not performing frame rate control; instead, the video output module (VO) will perform frame rate control. - User Picture When the network connection is abnormally disconnected and no stream is being sent from the front end, users can insert a user picture to be displayed on the VO to indicate the current network abnormality or that there is no stream to decode. VDEC provides two ways to insert a user picture: - Insert user picture immediately: VDEC first clears the stream and images inside the decoder, then inserts the user picture. - Insert user picture with delay: VDEC first decodes all streams inside the decoder, and only after all decoded images have been output does it insert the user picture. - Decoded Frame Buffer Allocation Mode - Decode Private VB Pool: When creating a decoding channel, VDEC creates a private VB pool as the image buffer for that channel. Users can set the number of private VB pool blocks frame\_buf\_cnt and the VB block size frame\_buf\_size in the channel creation interface [ss\_mpi\_vdec\_create\_chn](#ZH-CN_TOPIC_0000002408260210). - Decode User VB Pool: When creating a decoding channel, the image Buffer is not allocated. Instead, the user calls the ss\_mpi\_vb\_create\_pool interface to create a video cache VB pool, and then calls the [ss\_mpi\_vdec\_attach\_vb\_pool](#ZH-CN_TOPIC_0000002408260194) interface to bind a decoding channel to a specific video cache VB pool. The three decoded frame buffer allocation modes can be set via the parameter vb\_src of the [ss\_mpi\_vdec\_set\_mod\_param](#ZH-CN_TOPIC_0000002441659569) interface. When using Module VB pool or User VB pool mode, the VB pool can be destroyed without destroying the decoding channel first. However, before destroying the decoded VB pool, the user must ensure that no module is using any VB block in this pool (this can be done by resetting the decoding channel and the directly bound downstream modules, e.g., if VDEC is bound to VPSS, both VDEC and VPSS must be reset; if the user retrieves images from VDEC, all images must be released back to VDEC), otherwise program exceptions may occur. - Stream Buffer Configuration Mode Decoding stream buffer configuration supports two modes: normal mode and memory-saving mode. - Normal mode: The total stream buffer size is the user-configured stream\_buf\_size + channel width x channel height / 2. The minimum configurable value for stream\_buf\_size is channel width x channel height x 3/4. In this mode, the stream buffer can accommodate each frame of stream, including very large frames. - Memory-saving mode: The total stream buffer size is the stream\_buf\_size configured when the user creates the decoding channel. The minimum configurable value for stream\_buf\_size is 32KB. However, the user must ensure that the size of each frame sent to the stream does not exceed stream\_buf\_size. Memory-saving mode is invalid when sending streams in stream mode. The two modes can be set via the parameter mini\_buf\_mode of the [ss\_mpi\_vdec\_set\_mod\_param](#ZH-CN_TOPIC_0000002441659569) interface. mini\_buf\_mode=1 indicates memory-saving mode; mini\_buf\_mode=0 indicates normal mode. ## Module Parameters - g\_vdec\_max\_chn\_num The maximum number of channels supported by the VDEC module.
