--- # Audio ## Overview
The AUDIO module includes four sub-modules: audio input, audio output, audio encoding, and audio decoding. The audio input and output modules implement audio input/output functions by controlling the chip's audio interface. The audio encoding and decoding modules provide audio encoding/decoding capabilities for G711, G726, and ADPCM formats, and support recording and playing raw audio files in LPCM format. **Note: Unless otherwise specified, the description for is consistent with ; the description for is consistent with .** # Feature Description

## Audio Input and Audio Output ### Audio Interface and AI, AO Devices The audio input/output interface is referred to as the AIO (Audio Input/Output) interface. It is used to interface with the Audio Codec to accomplish sound recording and playback. AIO interfaces are divided into two types: input-only and output-only. - When of input type, it is also called AIP;[¶](#audio-input-and-audio-output-audio-interface-and-ai-ao-devices-the-audio-inputoutput-interface-is-referred-to-as-the-aio-audio-inputoutput-interface-it-is-used-to-interface-with-the-audio-codec-to-accomplish-sound-recording-and-playback-aio-interfaces-are-divided-into-two-types-input-only-and-output-only-when-of-input-type-it-is-also-called-aip "锚链接")

- When of output type, it is also called AOP. In software, the unit responsible for abstracting the audio interface input function is called an AI device; the unit responsible for abstracting the audio interface output function is called an AO device. For each input/output interface, the software establishes mapping relationships with AI devices and AO devices based on the functions supported by the interface. For example: AIP0 only supports audio input, so AIP0 maps to AiDev0; AOP0 only supports audio output, so AOP0 maps to AoDev0. ### Recording and Playback Principle Raw audio signals, given as analog signals, are converted to digital signals through the Audio Codec at a certain sample rate and sample bit width. The Audio Codec transmits the digital signals to the AI device using I2S timing or PCM timing. The chip uses DMA to transfer audio data from the AI device to memory, completing the recording operation. Playback is based on the same principle. The chip uses DMA to transfer data from memory to the AO device. The AO device sends data to the Audio Codec using I2S timing or PCM timing. The Audio Codec completes the conversion from digital signals to analog signals and outputs the analog signals. **Figure 1** Recording and Playback Schematic [](figures/Recording and Playback Schematic) ### Audio Interface Timing and AI, AO Channel Arrangement ### Audio Interface Timing The audio interface supports standard I2S interface timing mode and PCM interface timing mode, and provides flexible configuration to support interfacing with various Audio Codecs. For detailed timing support, please refer to the corresponding chip's user guide. To achieve successful interfacing, sufficient understanding of the I2S or PCM protocol and the timing support of the target Audio Codec is required. Below is a brief introduction to several characteristics of I2S and PCM interface timing: - According to standard I2S or PCM protocol, the most significant bit (MSB) is always transmitted first, followed by the least significant bit (LSB), i.e., serial data is transmitted from high bit to low bit.
- AI devices support extended multi-channel I2S and PCM interface timing for reception. When interfacing, the Codec's timing mode selection, sync clock, sample bit width, and other configurations must be consistent with the AI device's configuration, otherwise correct data may not be captured.
- AI/AO devices support master mode and slave mode. In master mode, the AI/AO device provides the clock. In slave mode, the Audio Codec provides the clock. In master mode, if the AI device and AO device interface with the same Codec simultaneously, the clock is shared for both input and output; there is no such restriction in other cases. In slave mode, the input and output clocks can be provided separately by peripheral Audio Codecs.
- Due to timing issues, in AI/AO slave mode, it is recommended that users configure the target Codec first, then configure the AI or AO device. In AI/AO master mode, it is recommended that users configure the AI or AO device first, then configure the target Codec.
- When AI/AO devices are in master mode, some AI/AO devices only provide frame sync clocks and bit clocks for timing synchronization, without providing MCLK. In this case, if the Audio Codec uses an external crystal oscillator as its working clock, it may cause sound distortion. Therefore, slave mode or using the bit clock to generate the Codec's internal master clock is recommended.
- When AI/AO devices are in master mode and provide MCLK externally, the MCLK settings for channel counts of 1/2/4/8/16 are: - For sample rates of 96k/48k/24k/12k, provide a 12.288MHz master clock. - For sample rates of 64k/32k/16k/8k, provide an 8.192MHz master clock. - For sample rates of 44.1k/22.05k/11.025k, provide an 11.2896MHz master clock. For channel count of 20, the MCLK settings are: - For sample rates of 48k/24k/12k, provide a 15.36MHz master clock. - For sample rates of 32k/16k/8k, provide a 10.24MHz master clock. - For sample rates of 44.1k/22.05k/11.025k, provide a 14.112MHz master clock. - AI/AO devices support standard PCM mode and custom PCM mode. PCM mode only supports mono mode. Therefore, in PCM mode, the right channel output of the target Codec must be disabled, otherwise the sound will have noise. According to the PCM standard mode protocol, when the working mode is standard PCM master mode or standard PCM slave mode, the audio input/output data is delayed by 1 bit clock (BCLK) cycle relative to the frame sync signal. If interfacing with an external Codec in standard PCM master mode or standard PCM slave mode, ensure that the number of bit clock (BCLK) cycles by which the external Codec's data is delayed relative to the frame sync signal is greater than the number of bit clock cycles (1 BCLK) by which the audio input/output data is delayed relative to the frame sync signal. Otherwise, the sound will have noise. **Figure 1** I2S 1/2/4/8/16/20-channel Reception [](figures/I2S 1-2-4-8-16-20 Channel Reception) Note: m0\~m3 are 4 channels of mic data. 1ch mode is only supported when non-time-division multiplexing is used. ### AI, AO Channels AI and AO channels are software-level concepts. The concept of channels is explained below through examples. For example: - When an AI device uses multi-channel multiplexed I2S reception mode, the standard I2S protocol only has the concept of left and right channels. The AI device supports up to 128 bits of audio data per left/right channel. Assuming the Codec has multiplexing capability and can multiplex 16 channels of 16-bit sample audio data into 2 channels of 128-bit I2S left/right channel data, then the AI device can parse these 16 channels of audio data and refer to them as 16 audio channels.
- When an AO device uses multi-channel multiplexed I2S transmission mode, the standard I2S protocol only has the concept of left and right channels. The AO device supports up to 128 bits of audio data per left/right channel. Assuming the Codec has multiplexing capability and can multiplex 8 channels of 24-bit sample audio data into 2 channels of 128-bit I2S left/right channel data, then the AO device can parse these 8 channels of audio data and refer to them as 8 audio channels.
- When an AO device uses multi-wire I2S transmission mode, only the standard I2S protocol is supported, i.e., each data line supports up to 24 bits of data per left/right channel. Assuming 4 data lines are used, the left and right channel data on each data line can be considered as 2 channels of audio data. In this case, the AO device can transmit these 8 channels of audio data and refer to them as 8 audio channels. AI and AO can split AI and AO channels within the maximum bit range supported by the AI/AO device, according to the sample bit width, and sequentially treat them as AiChn0, AiChn1, etc., or AoChn0, AoChn1, etc. Only when the I2S or PCM timing configured for the AI/AO device is consistent with the timing configured for the Codec can correct audio data be received or transmitted. The process of aligning the timing configuration of the AI/AO device with that of the Codec is called interfacing. For example, configure AI as standard PCM slave mode, 8k sample rate, 16bit sample precision, 2 channels; configure the frame sync clock provided by the Codec to 8k, bit clock to 8k x 32bit, each sample point to 32 bits (the Codec supports multiplexing two 16-bit sample points into one sample point). The channel arrangement in PCM timing is shown in [Figure 1](#fig3150146215), and the channel arrangement in I2S timing multi-wire mode is shown in [Figure 2](#fig5981314020). **Figure 1** PCM Timing Transmission Schematic [](figures/PCM Timing Transmission Schematic) **Figure 2** I2S Timing Multi-Wire Transmission Schematic [](figures/I2S Timing Multi-Wire Transmission Schematic) On , the maximum number of AI channels supported is 20. In I2S mode, the maximum number of AO channels supported is 8 (each data line supports 2 channels, but limited by the number of I2S data lines, only 4 channels are effective at maximum). In PCM mode, the maximum number of AO channels supported is 1. In multi-channel scenarios, AI and AO treat the corresponding two channels in the channel arrangement as stereo input/output. - For AI, for example in [Figure 1](#fig3150146215), channels 0 and 4, 1 and 5, 2 and 6, 3 and 7 are treated as the left and right channels of a stereo pair. That is, there are four 16-bit sample precision stereo inputs. In this case, only left channels 0~3 should be operated on.
- For AO, for example in [Figure 2](#fig5981314020), channels 0 and 4, 1 and 5, 2 and 6, 3 and 7 are treated as the left and right channels of a stereo pair. That is, each data line carries one stereo signal, with a total of four stereo outputs. In this case, only left channels 0, 1, 2, and 3 should be operated on. The AI device and AO device in the AIO interface are independent of each other. If they interface with the same Codec simultaneously, the working modes configured for AI and AO must be consistent. For example, both AI and AO are configured as I2S slave mode. When AIO operates in master mode, the product of the channel count and sample bit width must be equal for both AI and AO to ensure that the sync clock sent by AIO to the Audio Codec is consistent for both AI and AO. If they do not interface with the same Codec simultaneously, there is no such restriction. When AIO operates in slave mode, the product of channel count and sample bit width can be different for AI and AO, as the Audio Codec can send different sync clocks to AI and AO. The SDK supports establishing binding relationships between AI and AO channels through the SYS module binding interface, enabling real-time audio data playback. ### Resampling The audio input and output modules support resampling of audio data. If AI resampling is enabled, before [ss\_mpi\_ai\_get\_frame](#ZH-CN_TOPIC_0000002408099390) returns data, the internal resampling processing will be performed first, and then the processed data will be returned. If AO resampling is enabled, before audio data is sent to AO, internal resampling processing is performed first, and then the processed data is sent to the AO channel for playback. Audio resampling supports resampling between any two different sample rates (except 64k and 96k), and also supports downsampling from 64kHz to 8kHz or 16kHz. Supported input sample rates for resampling: 8kHz, 11.025kHz, 12kHz, 16kHz, 22.05kHz, 24kHz, 32kHz, 44.1kHz, 48kHz, 64kHz. Unsupported input sample rate: 96kHz. Supported output sample rates: 8kHz, 11.025kHz, 12kHz, 16kHz, 22.05kHz, 24kHz, 32kHz, 44.1kHz, 48kHz. Unsupported output sample rates: 64kHz, 96kHz. Resampling only supports processing mono data. - For AI resampling, the resampling input sample rate is the same as the sample rate configured in the AI device attributes. The resampling output sample rate must be different from the sample rate configured in the AI device attributes. The user only needs to configure the resampling output sample rate. The number of sample points per frame before resampling is the same as the number of sample points per frame configured in the AI device attributes.
- For AO resampling, the resampling output sample rate is the same as the sample rate configured in the AO device attributes. The resampling input sample rate must be different from the sample rate configured in the AO device attributes. The user only needs to configure the resampling input sample rate. The number of sample points per frame in the audio frame after resampling is the same as the number of sample points per frame configured in the AO device attributes. > **Notice:** >- When the AI channel mode is set to OT\_AI\_CHN\_MODE\_FAST and the data transmission method between AI-AO and AI-AENC is system binding mode, AI resampling is invalid. VQE addresses the similarities and differences between the AI and AO paths by using two scheduling logic modules, UpVQE and DnVQE, to process data on the two paths. - VQE currently supports four methods: TalkVQE, HIFIVQE, RecordVQE, and TalkVQEV2. - TalkVQE is mainly used for *DVR* voice intercom scenarios, suitable for low-quality voice processing (8/16kHz sample rate, 16bit width, mono), supporting two-way intercom scenarios. - HIFIVQE is mainly used for Mobile Camera recording scenarios, suitable for high-quality voice processing (48kHz sample rate, 16bit width, mono), and does not support two-way intercom. - RecordVQE is an upgraded version of HIFIVQE, supporting 16/48kHz sample rates, 16bit width, mono/stereo. For solutions that support RecordVQE and other methods, RecordVQE is recommended for Mobile Camera recording scenarios. - TalkVQEV2 is an upgraded version of TalkVQE, supporting 16kHz sample rate, 16bit width, stereo. For solutions that support TalkVQEV2 and other methods, TalkVQEV2 is recommended for *DVR* scenarios. - The AI upstream VQE path supports up to three modes: TalkVQE, HIFIVQE, and RecordVQE. Its functions include nine processing modules: Acoustic Echo Cancellation, Speech Noise Reduction, Automatic Gain Control, High-Pass Filter, Recording Noise Reduction, Equalizer, Dynamic Range Compression, Parametric Equalizer, and High Dynamic Range, as shown in [Figure 1](#fig10208037175514). The AI upstream VQE function is implemented in the [ss\_mpi\_ai\_get\_frame](#ZH-CN_TOPIC_0000002408099390) interface and returns processed data.
- The AO downstream VQE path supports the TalkVQE mode. Its functions include four processing modules: Speech Noise Reduction, Automatic Gain Control, High-Pass Filter, and Equalizer, as shown in [Figure 2](#fig1292512568568). The AO downstream VQE function is implemented in the [ss\_mpi\_ao\_send\_frame](#ZH-CN_TOPIC_0000002408259474) interface. VQE is divided into 2 scheduling interface modules (UpVQE and DnVQE), 11 function modules (AEC, ANR, AGC, RNR, EQ, HPF, GAIN, RES, HDR, DRC, PEQ), and 1 shared module (COMMON). The two scheduling logic modules dynamically load each function module using the `dlopen` method of the libdl library, and perform unified scheduling through a unified API interface format for function modules. VQE function modules support trimming.
