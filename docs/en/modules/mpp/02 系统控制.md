## Overview System Control, based on chip characteristics, completes the reset and basic initialization of each hardware component. It is also responsible for initializing and de-initializing each business module of the MPP (Media Process Platform) system, managing the working status of each MPP system business module, providing the current MPP system version information, and providing large-block physical memory management functions. Before an application starts MPP services, it must complete MPP system initialization. Similarly, after the application exits MPP services, it must complete MPP system de-initialization to release resources. > **Notice:** ## Important Concepts ### Low Latency Low latency means that after the image is written to the specified number of lines to DDR, the front-end module reports an interrupt and sends the image to the back-end module for processing. This can reduce pipeline latency, and the front-end and back-end modules use a specific mechanism to ensure that the image is written before being read, so no image read errors occur. > **Notice:** >- Enabling low latency on each module may conflict with certain specific feature combinations. ## Functional Description ### Video Buffer Pool The video buffer pool mainly provides large-block physical memory management functions for media services, responsible for memory allocation and recycling, fully leveraging the role of the memory buffer pool to ensure reasonable use of physical memory resources across various media processing modules. A group of cache blocks of the same size and contiguous physical addresses form a video buffer pool. The public video buffer pool must be configured before system initialization. Depending on the service, the number of public buffer pools and the size and quantity of cache blocks may vary. All video input channels can obtain video cache blocks from the public video buffer pool to store captured images. As shown in [Figure 1](#fig17183122371915), VI obtains cache block Bm from the public video buffer pool B. Cache block Bm is sent from VI to VPSS. After being processed by VPSS, the input cache block Bm is released back to the public video buffer pool. Assuming the VPSS channel working mode is USER, VPSS channel 0 obtains cache block Bi from public video buffer pool B as the output image cache buffer and sends it to VENC. VPSS channel 1 obtains cache block Bk from public video buffer pool B as the output image cache buffer and sends it to VO. Bi is released back to the public video buffer pool after VENC encoding is complete, and Bk is released back after VO display is complete. **Figure 1** Typical Public Video Buffer Pool Data Flow Diagram 【describe】 For different types of video buffer pool size calculation, see the code in ot\_buffer.h >[](../../../../multimedia/mpp/public_sys-resources/icon-notice.gif) **Notice:** >When Hi3403V100 allocates the buffer required for MCF scenarios, if the mono path is not a luminance single-component image, the ot\_vpss\_get\_mcf\_color\_buf\_size must be used to calculate the buffer size. **Table 1** Introduction to Video Buffer Pool Size Calculation Interfaces in ot\_buffer.h

### Forced VB Destruction When VB is in an occupied state, it cannot be destroyed. By adding the module parameter g\_vb\_force\_exit=1 when inserting xx\_base.ko, VB can be forcibly destroyed even when it is in use. Therefore, use this feature with caution. Ensure that VB is not actively destroyed during normal service operation. VB must only be actively destroyed after all services have completely exited. >[](../../../../multimedia/mpp/public_sys-resources/icon-notice.gif) **Notice:** >The forced VB destruction feature is designed to facilitate service restart after an application crashes abnormally. After a user-mode application crashes abnormally, the application can no longer follow the normal service exit procedure.
