--- # Video Input

## Overview The Video Input (VI) module receives video data through MIPI Rx (including MIPI, LVDS, and HISPI interfaces), BT.1120, BT.656, BT.601, DC, and other interfaces. VI stores the received data into specified memory regions. During this process, VI can process the received raw video image data to implement video data capture. ## Important Concepts - Video Input Device A Video Input Device supports several timing input types and is responsible for timing parsing. - Video Input Physical PIPE A Video Input PIPE is bound to the back end of a device and is responsible for further processing of data parsed by the device. - Video Input Virtual PIPE A Video Input Virtual PIPE is not bound to a device and is responsible for further processing of data sent by other modules or the user. - Video Physical Channel A Physical Channel is responsible for outputting the final processed data to DDR. Before actually outputting data to DDR, it can implement functions such as cropping. - PIPE Working Mode See"System Control"Chapter"V Iand VPSS"Working Mode Description. - Mask The mask is used to indicate the video data source of a VI device. - Lens Distortion Correction (LDC) Lens Distortion Correction: Some low-end lenses tend to produce image distortion, requiring the image to be corrected based on the degree of distortion. - DIS The DIS module compares the current image with the previous two frames using stabilization algorithms of varying degrees of freedom, calculates the shake offset vector of the current image in each axis direction, and then corrects the current image based on the shake offset vector, thereby achieving stabilization. - BAS Bayer scaling, i.e., Bayer domain scaling. - Early Interrupt Reporting Early interrupt reporting means that after a specified number of image lines have been written to DDR, VI reports an interrupt and sends the image to the back-end module for processing. This can reduce latency, but unlike Low Latency mode, there is no hardware mechanism to ensure that the back-end module will read the image without errors. ## functional Description ### functional Block Diagram **Figure 1** The Four Parts of VI at the Software Level VI is divided at the software level into four layers: Input Device (DEV), Input PIPE (illustrated as Physical PIPE; Virtual PIPE contains only ISP\_BE), Physical Channel (PHY\_CHN), and Extension Channel (EXT\_CHN). The user needs to determine which slave mode module to use based on the SENSOR pin connections and [Table 1](#_Ref486405463), then select the corresponding Physical PIPE number to create a Physical PIPE; otherwise there will be no data. If modification is needed, it can be done by modifying ISP-related code. ### Mask Configuration The upper 12 bits of the mask correspond to the 12 pin connections on the hardware circuit (any 12 consecutive pins between D0 and D15, e.g., D4\~D15). The user must set the appropriate mask configuration based on the actual connection. The most significant bit of the mask corresponds to pin D15. For example, if a 10-bit input sensor is connected to pins D6\~D15, the mask is configured as 0x FFC00000. Similarly, for 14-bit input, the corresponding mask is configured as 0x FFFC0000. The VI data line order is from low to high. For example, for single-component input, D0 is the low bit of data and D15 is the high bit of data. - 1-channel/2-channel 5M or 1080p Image Input Scenario (12-bit Input) In 1-channel/2-channel 5M or 1080p image input scenarios, when setting VI device attributes, the mask can be configured according to [Table 1](#_Ref331669300). **Table 1** Mask Configuration for 1-channel, 2-channel 5M or 1080p Scenarios (12-bit)
