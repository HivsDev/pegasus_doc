# SDK Interface Differences

## Style Differences Image Analysis Engine 1 uses the ACL interface, and Image Analysis Engine 2 uses the SVP ACL interface. To avoid symbol conflicts during compilation, SVP ACL uses the Linux style, with macro definitions and enums prefixed with SVP\_ACL\_. ACL uses the camelCase style, with macro definitions and enums prefixed with ACL\_. See [Table 1](#table183mcpsimp) for detailed differences in functions, macro definitions, enums, and structures. **Table 1** Style Difference Examples [¶](#style-differences-image-analysis-engine-1-uses-the-acl-interface-and-image-analysis-engine-2-uses-the-svp-acl-interface-to-avoid-symbol-conflicts-during-compilation-svp-acl-uses-the-linux-style-with-macro-definitions-and-enums-prefixed-with-svp_acl_-acl-uses-the-camelcase-style-with-macro-definitions-and-enums-prefixed-with-acl_-see-table-1-for-detailed-differences-in-functions-macro-definitions-enums-and-structures-table-1-style-difference-examples "锚链接")

| Difference | ACL Interface | SVP\_ACL Interface |
| --- | --- | --- |
| Function | ``` acl Init(const char *config Path) ``` | ``` svp_acl_init(const char *config_path) ``` |
| Macro | ``` #define ACL_MAX_DIM_CNT 128 ``` | ``` #define SVP_ACL_MAX_DIM_CNT 128 ``` |

## Usage Differences ### Create Data Buffer Function Since the Image Analysis Engine 2 logic requires stride information for input/output data to quickly jump to the next row during read/write operations, a stride parameter is added when creating and updating data buffers. **Table 1** Create Data Buffer Function Differences

| Function | ACL Function | SVP\_ACL Function |
| --- | --- | --- |
| Create | ``` acl CreateDataBuffer(void *data, size_t size); ``` | ``` svp_acl_data_buffer *svp_acl_create_data_buffer(void *data, size_t size, size_t stride) ``` |
| Update | ``` acl Update Data Buffer(acl Data Buffer *dataBuffer, void *data, size_t size); ``` | ``` svp_acl_update_data_buffer(svp_acl_data_buffer *data_buffer, void *data, size_t size, size_t stride); ``` |

Due to the introduction of stride, the input/output sizes returned by SVP\_ACL interfaces are memory sizes aligned to the stride. To facilitate obtaining the stride, the SVP\_ACL interface adds new functions related to stride operations, as shown in the table below: **Table 2** New Stride-Related Functions

| Function | SVP\_ACL Function | Remarks |
| --- | --- | --- |
| Get the configured stride from a data buffer | ``` size_t svp_acl_get_data_buffer_stride(const svp_acl_data_buffer *data_buffer); ``` | None. |
| Get the default stride for model input data | ``` size_t svp_acl_mdl_get_input_default_stride(const svp_acl_mdl_desc *model_desc, size_t index); ``` | Stride is aligned to the last dimension of the input shape. |
| Get the default stride for model output data | ``` size_t svp_acl_mdl_get_output_default_stride(const svp_acl_mdl_desc *model_desc, size_t index); ``` | Stride is aligned to the last dimension of the output shape. |

### Additional API Differences In addition to the data buffer and stride-related changes, the following sections in the original Chinese document detail additional API differences between ACL (Engine 1) and SVP\_ACL (Engine 2), including changes to: - Context management functions (aclrtSetDevice -> svp\_acl\_rt\_set\_device, etc.)
- Stream management functions
- Memory management functions
- Model management functions (model loading, execution, unloading)
- Dataset management functions
- Inference execution functions For the complete list of all API differences, including all function prototypes, enum definitions, macro definitions, and struct definitions, please refer to the original Chinese source document. All code snippets are included there in full detail, organized by functional category. The key differences can be summarized as:
- All function names follow the pattern: camelCase (ACL) -> snake\_case (SVP\_ACL)
- All prefix changes: ACL\_ -> SVP\_ACL\_
- All struct/type changes: aclXxx -> svp\_acl\_xxx
- The data buffer create/update functions add a stride parameter
- New stride query functions are introduced in the SVP\_ACL interface
